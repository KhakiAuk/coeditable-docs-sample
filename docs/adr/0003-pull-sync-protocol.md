# ADR-0003: プル型同期プロトコルの採用

- **ステータス**: Accepted
- **日付**: 2025-03
- **関連**: [アーキテクチャ詳細](../architecture.md), [ADR-0002](./0002-bidirectional-two-channels.md)

---

## コンテキスト

DataChannel が open になった時点で相手の recvCh がまだ subscribe 前、という **タイミング競合**が発生しうる。プッシュ型では open 直後に送ったメッセージがドロップされるケースが生じる。

## 決定

**プル型（リクエスト/レスポンス）** の同期プロトコルを実装する。

1. `sendCh.open` になったら `MSG_SYNC_REQ` を送る
2. 相手の `recvCh` が open になった時点で初めてメッセージを処理できる
3. 相手は `MSG_SYNC_STATE`（フル状態）で応答する
4. `recvCh.open` 後のリカバリとして、`recvCh.open` 時にも `MSG_SYNC_REQ` を再送する

### メッセージタイプ

| タイプ           | バイト | 内容                                           |
| ---------------- | ------ | ---------------------------------------------- |
| `MSG_SYNC_REQ`   | `0x00` | 「あなたの現在状態を送ってください」リクエスト |
| `MSG_SYNC_STATE` | `0x01` | フル Yjs 状態（`Y.encodeStateAsUpdate`）の応答 |
| `MSG_UPDATE`     | `0x02` | インクリメンタルな Yjs 更新                    |

## 理由

- open のタイミング競合を完全に解消できる
- リクエスト側は相手が open になるまで待たずに送信でき、相手は受け取った時点で応答できる
- 3 種類のメッセージタイプという最小プロトコルで実装できる

## トレードオフ

- フル状態の転送が初回に発生するため、ドキュメントサイズが大きい場合は初期同期に時間がかかる
- `MSG_SYNC_REQ` の再送（`recvCh.open` 時）により、フル状態が複数回送られる可能性がある（Yjs の `applyUpdate` は冪等なので問題なし）
